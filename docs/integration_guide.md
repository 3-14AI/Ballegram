# Руководство по интеграции клиентов Ballegram

Настоящее руководство предназначено для разработчиков клиентских приложений (мобильных, веб) для Ballegram.

## 1. Архитектура интеграции

Клиентское приложение общается с бэкендом Ballegram по двум основным протоколам:

1.  **HTTP/REST:** Используется для большинства операций: регистрация, авторизация, профиль, загрузка ленты, поиск пользователей, загрузка медиафайлов.
2.  **WebSocket:** Используется для функциональности чатов в реальном времени.

## 2. Управление состоянием (State Management)

Для эффективной работы клиента рекомендуется использовать глобальное хранилище состояния (например, Redux/Zustand в React, BLoC/Provider в Flutter).

*   **Auth State:** Хранит текущий JWT токен и информацию о пользователе. Если токен истекает (401 Unauthorized), клиент должен перенаправить пользователя на экран входа.
*   **Chat State:** Поддерживает активное WebSocket соединение. Хранит кэш списка чатов и последних сообщений.
*   **Feed State:** Кэширует посты для офлайн-просмотра или быстрого переключения между экранами.

## 3. Обработка ошибок

*   Все HTTP-ответы с кодом `4xx` и `5xx` должны обрабатываться на клиенте.
*   Ответы API содержат JSON с описанием ошибки, если это применимо.
*   Особое внимание следует уделить обработке ошибок сети и разрывам WebSocket-соединения:
    *   Клиент должен реализовывать логику автоматического переподключения (reconnection) с экспоненциальной задержкой.
    *   При обрыве связи сообщения должны ставиться в локальную очередь и отправляться при восстановлении.

## 4. Пагинация и кэширование

*   Для загрузки ленты (Feed) и истории чата клиенту следует использовать параметры пагинации (обычно `limit` и `offset` или `cursor`).
*   Для снижения нагрузки на сервер и экономии трафика рекомендуется кэшировать медиафайлы (изображения) на устройстве пользователя.

## 5. Безопасность

*   JWT токен должен храниться в безопасном месте на устройстве (например, Keychain для iOS, EncryptedSharedPreferences для Android, HttpOnly cookies или Secure Storage для веб).
*   Пароли при регистрации и входе должны передаваться по защищенному протоколу (HTTPS).

## 6. Загрузка медиа (Media Uploads)

Процесс загрузки изображения:
1.  Пользователь выбирает файл.
2.  Клиент сжимает/оптимизирует файл перед отправкой (если требуется).
3.  Клиент отправляет `POST` запрос (multipart/form-data) на эндпоинт загрузки Ballegram.
4.  Сервер возвращает URL загруженного файла (например, на S3/MinIO).
5.  Клиент использует этот URL для прикрепления к сообщению или посту.

## 7. Развертывание локальной среды разработки

Для тестирования клиента локально:
1.  Установите Docker.
2.  Запустите инфраструктуру Ballegram (БД, Redis) через `docker-compose up -d`.
3.  Запустите бэкенд сервер Ballerina: `bal run`.
4.  Настройте клиентское приложение на использование `http://localhost:9090` в качестве `baseUrl`.
